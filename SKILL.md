# A股期权持仓解析助手

当用户要求解析 A 股期权持仓明细时，请严格按照以下工作流执行：

## 0. 获取截图数据
- 如果用户直接在对话中附带了截图，请直接解析该截图。
- 如果用户**没有指定或上传截图文件**，请默认前往 `~/Downloads/options/` 目录下查找最新的期权持仓截图文件进行解析。

## 1. 数据提取 (OCR & 结构化提取)
仔细识别并提取截图中的每一行持仓记录。**不要在回复中直接输出长篇 Markdown 表格**，而是将提取的数据结构化为标准的 JSON 格式。

必须提取并清洗以下字段（请根据截图内容准确映射）：
- `underlying`: 持仓标的（例如：“创业板ETF”、“500ETF”、“中证1000”等，从合约名称中剔除“购”、“沽”及后续部分）
- `expiry_date`: 到期时间。如果截图直接提供“YYYY-MM-DD”格式则直接提取；如果只提供了月份和“剩余天数”（如“22天”），请以当前解析时的自然日为基准，加上剩余天数推算精确的“YYYY-MM-DD”日期（如：当前是2026-06-25，剩余22天，则推算为2026-07-17）。
- `option_type`: 期权类型（识别合约名称中的“购”或“沽”，分别映射为 `"CALL"` 或 `"PUT"`）
- `strike_price`: 行权价（提取合约名称中的行权价数值，例如：4600、9500）
- `current_price`: 现价（浮点数）。**注意**：如果是股指期权，截图中可能只显示当前选中合约的现价。对于缺失现价的行，可通过公式反推：`现价 = 开仓均价 - 逐笔盈亏 / (持仓数量 * 合约乘数)`。
- `open_price`: 买入价/开仓价（浮点数）
- `pnl_amount`: 盈亏金额/浮动盈亏（浮点数，注意保留正负号）
- `quantity`: 持仓数量（整数）。**关键判定逻辑**：观察明细表该行中是否包含**“保”**字标识，或**“类型”**列是否为**“卖”**：
  - 如果包含“保”字或类型为“卖”，说明该笔持仓是保证金交易（卖出期权），此时提取的持仓数量**必须转换为负数**（例如截图中数字为8，则提取为 `-8`）。
  - 如果不包含且类型为“买”，则是买入期权，持仓数量为正数。

**数据校验与A股特性说明**：
A股期权有两类主要的合约乘数：
- **ETF期权**（如50ETF、创业板ETF）：合约乘数通常为 `10000`。
- **股指期权**（如中证1000、沪深300股指期权）：合约乘数通常为 `100`。
无论买方还是卖方（正数或负数），其浮动盈亏的通用计算逻辑均为：
`pnl_amount ≈ (current_price - open_price) * quantity * 合约乘数`
提取过程中，可以在内部严格校验提取出的 `open_price`、`current_price`、带有正负号的 `quantity` 和 `pnl_amount` 是否符合该运算逻辑。如果发现偏差过大（可能由于图片模糊导致 OCR 错误），请先暂停，并在回复中与用户核对异常数据行。

## 2. 输出 JSON 数据
校验无误后，将所有数据格式化为 JSON 数组，并将该 JSON 内容直接写入到本地文件中（注意根据当天日期创建对应目录）：
`~/outputs/a-share-option-position-parser-skill/<YYYY-MM-DD-描述>/position_data.json`

**特别注意（覆写机制）**：
每次执行此技能时，必须生成一份全新的数据。在执行文件写入时，**如果该路径下已经存在旧的 `position_data.json` 文件，请务必直接覆写（Overwrite）或将其删除后重新生成**，绝不能追加写入。

JSON 格式示例：
```json
[
  {
    "underlying": "创业板ETF",
    "expiry_date": "2026-12-23",
    "option_type": "CALL",
    "strike_price": 4600,
    "current_price": 0.2598,
    "open_price": 0.2676,
    "pnl_amount": 78.00,
    "quantity": -1
  }
]
```

## 3. 自动导入到系统 (可选配置)
请检查本 Skill 目录下（`~/.gemini/config/skills/a-share-option-position-parser-skill/config.json`）的配置文件：
- 如果存在该文件且配置了 `api_import_url`（如 `"api_import_url": "http://127.0.0.1:5001/api/a_etf_option_positions/import"`），则必须使用 `curl` 工具（或 Python 脚本）将这份 JSON 数组数据自动推送到该接口完成导入：
  - **请求方法**: `POST`
  - **请求头**: `Content-Type: application/json`
  - **请求体**: 刚刚生成的完整 JSON 数组数据
- 如果**不存在**该文件或未配置 `api_import_url`，则**跳过此导入步骤**。

**示例命令（如果已配置）**：
```bash
curl -X POST <读取到的_api_import_url> \
-H "Content-Type: application/json" \
-d @/绝对路径/到/你的/position_data.json
```

## 4. 回复用户
保存 JSON 文件后，用 1-2 句话向用户总结说明“截图数据已成功解析（保存至 xxx.json）”。
如果执行了 API 导入步骤，请一并告知导入结果（成功或失败的原因及条数）。如果未配置导入接口而跳过了，则无需提及导入。
